# BACKEND_STRUCTURE_PLAN.md

配套文档：`FEATURE_REQUIREMENTS.md`、`DEV_TASK_BREAKDOWN.md`、`DATA_MODEL_DRAFT.md`

本文用于定义本项目更适合长期演进的后台结构方案，目标是把当前“按表平铺的 CRUD 菜单”逐步收敛成“按业务流程组织的运营后台”。

## 一、为什么现在要先定后台结构

当前 discover 插件后台是 FastAdmin 生成式 CRUD 结构，优点是落地快，缺点是：

- 菜单按数据表平铺，不按运营任务组织。
- 学校管理员真正高频的工作只有少数几项，但菜单入口很多。
- `关注 / 点赞 / 消息提醒 / 收藏` 这类数据更像行为明细，不适合作为长期主菜单。
- 后续一旦接入学校隔离、广告、群二维码、同步配置，后台会越来越分散。

因此，在继续开发第二阶段前，先明确“目标后台信息架构”是合理的，这样后续功能不会越做越散。

## 二、当前后台结构问题

### 当前 discover 菜单现状

当前菜单基本是以下结构：

- 发现-关注
- 发现-评论
- 发现-动态
- 发现-点赞
- 发现-标签
- 发现-话题
- 发现-消息提醒
- 学校管理（新加）

### 当前结构的主要问题

- 以表为中心，不以任务为中心。
- 学校管理员看到很多低价值菜单，容易误解哪些是日常必用功能。
- 后续不同学校的数据虽然可以在查询上隔离，但菜单语义仍然很混乱。
- 运营动作和内容治理动作混在一起，不利于扩展广告、群码、同步配置等能力。

## 三、目标后台设计原则

### 1. 业务驱动，而不是表驱动

后台菜单应围绕“管理员要完成什么任务”组织，而不是围绕“数据库里有哪些表”组织。

### 2. 学校管理员只看到必要功能

子管理员的核心职责是：

- 审核本校帖子
- 置顶本校帖子
- 删除本校帖子
- 管理本校评论（如需要）

因此，子管理员后台必须尽量精简。

### 3. 低价值明细不做主菜单

以下模块不建议长期作为一级或高频二级菜单：

- 关注
- 点赞
- 收藏
- 消息提醒

它们更适合：

- 挂在帖子详情页
- 挂在用户详情页
- 做成统计/只读页
- 或留给总账号做数据排查用途

### 4. 学校隔离必须在数据层实现

后台结构可以优化体验，但真正的隔离必须由服务端查询条件、写操作权限、数据归属字段来完成，不能只靠菜单隐藏。

### 5. 总账号与子账号看到的后台可以不同

同一套系统里，应该允许不同角色看到不同的菜单树和功能深度。

## 六、当前 discover 子菜单的去留建议

### 建议保留并升级为核心菜单

- `discover/discover`
  - 不建议继续叫“发现-动态”
  - 建议改成“帖子管理”或“内容管理”
- `discover/comment`
  - 建议保留，作为评论治理入口
- `discover/tag`
  - 建议保留，但更适合作为“内容配置”而不是日常操作入口
- `discover/topic`
  - 建议保留，但同样更适合作为配置入口
- `discover/school`
  - 必须保留，且放到更高优先级位置

### 建议降级，不作为长期主菜单

- `discover/attentions`
- `discover/favor`
- `discover/log`
- `discover/collect`（当前虽然菜单里是隐藏状态，但从结构上也不建议提升）

### 更合理的处理方式

这些低价值模块后续建议改为：

- 帖子详情里的只读明细
- 用户详情里的行为统计
- 数据中心下的只读查询页
- 总账号排查用途，不对学校管理员开放

## 七、学校数据隔离应该怎么做

### 1. 必须直接挂学校的对象

这些对象建议直接带 `school_id`：

- 帖子
- 学校广告
- 学校群二维码
- 子管理员绑定关系
- 同步目标配置（如果按学校区分）

### 2. 可通过主对象反推学校的对象

这些对象可以通过帖子或评论反查学校：

- 评论
- 点赞
- 收藏
- 消息提醒
- 关注（视业务而定，若是作品触发可间接归属）

### 3. 推荐的隔离规则

- 总账号：全量可见
- 子管理员：只看绑定学校数据
- 跨学校的写操作一律禁止
- 批量操作逐条校验学校归属

## 八、推荐的页面组织方式

### 1. 帖子审核页

目标：让学校管理员先做最核心的审核工作。

建议包含：

- 待审核列表
- 审核通过 / 驳回
- 置顶 / 取消置顶
- 按学校、时间、状态筛选

### 2. 帖子管理页

目标：管理已发布内容。

建议包含：

- 已通过帖子列表
- 已驳回帖子列表
- 已置顶帖子列表
- 删除操作
- 查看关联评论、点赞、收藏、消息提醒摘要

### 3. 评论管理页

目标：治理评论区，而不是把“消息提醒”当成管理入口。

建议包含：

- 评论列表
- 屏蔽 / 删除
- 查看归属帖子
- 查看归属学校

### 4. 数据中心页

目标：统一承载行为明细和统计，而不是把行为表直接暴露成运营菜单。

建议后续集中承载：

- 点赞排行
- 关注增长
- 内容互动统计
- 群二维码曝光/点击
- 广告曝光/点击

## 九、后台改造计划

## 阶段 1：先稳住结构，不大改路径

目标：不打断当前开发节奏，先保证功能能继续做。

- 保留现有 discover CRUD 代码结构。
- 优先把“学校管理、子管理员绑定、帖子学校隔离”跑通。
- 临时允许旧菜单存在，但逐步减少使用。

## 阶段 2：菜单收敛

目标：把后台从“表菜单”改成“业务菜单”。

建议动作：

- `发现-动态` 重命名为 `帖子管理`
- 新增 `帖子审核` 页面（可复用 discover 列表，默认筛待审核）
- `发现-评论` 重命名为 `评论管理`
- `学校管理` 提升到更清晰位置
- 隐藏 `关注 / 点赞 / 消息提醒 / 收藏` 的主菜单入口

## 阶段 3：增加运营中心

目标：承载后续首页广告、群二维码、同步配置。

建议动作：

- 新增 `运营中心`
- 将广告、群码、同步配置放入运营中心
- 学校管理员只看到本校配置，总账号看全局配置

## 阶段 4：增加数据中心

目标：把低价值明细迁移到只读数据查询场景。

建议动作：

- 将点赞、关注、消息提醒做成只读统计页或明细页
- 从日常运营菜单中降级出去

## 十、对当前开发顺序的建议

在现有第一阶段完成后，我建议后续顺序变成：

1. 完成学校与帖子归属的第一阶段闭环
2. 先做后台菜单与页面收敛
3. 再做小程序学校选择
4. 再做首页按学校展示
5. 再做群二维码和首页广告
6. 最后做发布同步与日志

原因是：

- 如果后台入口不先收敛，后面功能越加越多，管理体验会更差。
- 先明确总账号/子账号看到什么，再做前后端功能，后续返工会更少。

## 十一、我建议的最终结论

### 结论 1

当前 discover 的后台结构可以作为“过渡实现”，但不适合作为最终后台结构长期保留。

### 结论 2

最适合你这个项目的后台结构，不是“发现-关注 / 评论 / 动态 / 点赞 / 消息提醒”并排展开，而是：

- 学校运营
- 内容中心
- 运营中心
- 数据中心

### 结论 3

学校隔离的关键不是菜单本身，而是：

- `school_id` 归属模型
- 服务端查询条件
- 写操作权限校验
- 角色视图差异化显示

## 十二、下一步建议

如果你认可这份方案，后续我建议先做这两件事：

- 先新增一个更贴近业务的后台入口：`帖子审核`
- 然后把当前低价值 discover 菜单降级或隐藏

这样后面继续做学校选择、首页广告、群二维码时，后台就不会越做越乱。

## 十三、关于审核策略与“发现种草”拆分方式的补充说明

### 1. 帖子审核是否一定要人工逐条审核

不建议把“帖子审核”理解为“所有帖子都人工逐条审核”。

更现实的做法是：

- 保留审核入口
- 但只让异常内容、待处理内容、被举报内容进入审核池
- 正常低风险内容默认自动通过

因此，“帖子审核”页面更适合作为：

- 待审核内容处理台
- 风控复核台
- 举报复审台

而不是“所有内容的人肉审稿后台”。

### 2. 推荐的审核模式

建议采用“分层审核”而不是“全量人工审核”：

- 第一层：默认自动通过
- 第二层：命中规则后进入待审核
- 第三层：人工审核处理异常内容

后续可逐步增加以下触发条件：

- 新用户首帖
- 敏感词命中
- 图片或文本风险命中
- 被举报内容
- 某学校开启严格审核模式

### 3. “发现种草”该怎么拆分更合适

如果继续把“发现种草”下面直接铺很多小菜单，长期一定会越来越乱。

更推荐的拆法是：

- 先按大块拆
- 再在大块下面保留必要的小功能入口

也就是先有：

- 内容中心
- 运营中心
- 数据中心
- 学校运营

再在下面细分：

- 内容中心
  - 帖子审核
  - 帖子管理
  - 评论管理
  - 标签管理
  - 话题管理
- 运营中心
  - 首页广告
  - 群二维码
  - 同步配置
- 数据中心
  - 点赞/关注/消息提醒明细（只读）
  - 统计看板
- 学校运营
  - 学校管理
  - 子管理员管理

### 4. 什么时候保留“发现种草”作为一个大菜单

如果当前后台的主业务几乎全部都属于“发现种草”，那么短期内可以保留它作为一个一级菜单。

但建议它下面只挂“大块入口”，而不要继续挂很多表级菜单。

更推荐的短期结构是：

- 发现种草
  - 帖子审核
  - 帖子管理
  - 评论管理
  - 标签管理
  - 话题管理
  - 学校管理

也就是说：

- 短期：`发现种草` 可以保留为一级菜单
- 中期：其下只保留高价值二级菜单
- 长期：如果平台业务继续增大，再把“内容中心 / 运营中心 / 数据中心 / 学校运营”提升为一级菜单

### 5. 当前最适合本项目的建议

基于你现在的项目阶段，我更推荐：

- 当前不要一次性把所有一级菜单打散
- 先保留一级菜单 `发现种草`
- 但二级菜单收敛成少数几个高价值入口
- 把低价值明细菜单逐步降级为只读页、详情页或统计页

这样既不会把当前后台改得过大，又能防止后续继续按表平铺失控。
