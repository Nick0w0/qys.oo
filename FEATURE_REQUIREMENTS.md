# FEATURE_REQUIREMENTS.md

本文用于沉淀当前已明确的功能需求，供后续产品拆解、开发评估、测试验收和联调参考。

## 一、需求概览

本期需求围绕“校园帖子管理、学校归属、社群引流、运营展示”展开，目标是：

- 完善总账号/子账号的后台管理边界。
- 建立“用户地理位置 -> 学校 -> 帖子内容”的归属关系。
- 强化帖子发布后的群聊/平台同步能力。
- 在小程序端增加进群引导与首页活动运营位。

## 二、角色定义

### 1. 总账号管理员

- 拥有平台级最高权限。
- 可查看和管理全部学校的数据。
- 可新增、编辑、禁用、删除子管理账号。
- 可为子管理账号分配所属学校和可操作权限。

### 2. 子管理账号

- 仅能管理被分配学校的数据。
- 仅能执行帖子审核、置顶、删除等授权动作。
- 不可查看、不可操作、不可关联其他学校的数据。

### 3. 普通用户

- 使用小程序浏览、发布、互动。
- 可基于地理位置选择学校。
- 进入小程序时可收到进群二维码引导。

## 三、功能需求清单

## 需求 1：子账号管理员只能管理本校帖子审核 / 置顶 / 删除，不能关联其他学校

### 业务目标

- 让学校运营管理职责分层，避免跨校误操作。
- 保证帖子审核权限只在所属学校内生效。

### 功能说明

- 系统一期支持为每个子管理账号绑定一个“所属学校”，确保权限边界清晰。
- 子管理账号登录后台后，仅可看到所属学校范围内的帖子数据。
- 子管理账号仅可对所属学校帖子执行以下操作：
  - 审核
  - 置顶
  - 删除
- 子管理账号不可新增、修改、删除其他学校信息。
- 子管理账号不可操作非所属学校的帖子。
- 子管理账号不可通过接口参数、搜索条件、导入导出等方式绕过学校限制。

### 后台要求

- 管理员账号需要增加“所属学校”配置项；技术实现按可扩展结构设计，后续可演进为多学校绑定。
- 帖子列表、帖子详情、审核接口、置顶接口、删除接口都必须校验学校归属。
- 后台查询条件、统计数据、筛选项均应按学校权限收口。

### 验收标准

- 子管理账号 A 绑定学校 X 后，只能看到学校 X 的帖子。
- 子管理账号 A 无法审核、置顶、删除学校 Y 的帖子。
- 即使手动篡改请求参数，也不能操作其他学校数据。
- 总账号可查看所有学校帖子并进行管理。

### 审核策略建议

- 建议保留“帖子审核”能力，但不默认采用全量人工逐条审核。
- 更现实的做法是分层审核：
  - 默认普通内容自动通过
  - 命中规则的内容进入待审核
  - 被举报或高风险内容进入人工复核
- 一期可以先支持以下触发待审核的条件：
  - 新用户首帖
  - 命中敏感词或风控规则
  - 学校开启严格审核模式
  - 被系统或用户举报的内容
- 后台“帖子审核”页面更适合定位为“待处理内容台 / 风控处理台”，而不是全部内容的人审中心。

## 需求 2：根据地理位置选择学校

### 业务目标

- 帮助用户快速定位所属学校，提高内容归属准确性。
- 为后续“按学校分发内容”和“学校管理权限控制”提供基础数据。

### 功能说明

- 用户进入小程序时，系统可请求定位权限。
- 系统根据用户地理位置推荐附近学校。
- 用户可从推荐学校中选择，也可手动搜索学校和按城市筛选学校。
- 用户一旦选择学校，后续发布帖子、浏览内容、互动记录均应带上学校归属。
- 每个用户只能绑定一个学校。
- 学校一旦确认绑定，普通用户不可自助修改；如确需修改，只能由后台人工处理。
- 用户在最终绑定前，必须经过明确确认。

### 前端要求

- 提供定位授权提示。
- 提供学校推荐列表。
- 提供学校搜索能力。
- 提供城市筛选能力。
- 在最终绑定前提供二次确认弹窗，明确提示“选择后将无法自行修改”。
- 在页面中明确展示当前所选学校。
- 已绑定后不再向普通用户提供自由切换入口。

### 后台要求

- 平台需具备学校数据源，至少包含：
  - 学校名称
  - 学校 ID
  - 所在城市 / 区域
  - 经纬度或可用于定位匹配的数据
- 帖子数据需要增加学校归属字段。
- 用户数据需要增加单一学校绑定字段或等价配置。
- 后台需提供人工修改用户学校的能力，并保留操作日志。

### 验收标准

- 用户授权定位后，可看到附近学校推荐。
- 用户拒绝定位时，仍可手动通过搜索和城市筛选选择学校。
- 用户确认学校前会看到“绑定后无法自行修改”的二次确认提示。
- 用户绑定成功后，普通用户端不可自行修改学校。
- 如需修改，只能由后台人工处理，并保留记录。
- 用户发布的帖子能够正确绑定学校。
- 首页内容可以根据当前学校进行筛选或展示。

## 需求 3：群聊 / 平台里面发布同步更新

### 业务目标

- 提高内容分发效率。
- 帖子发布后可同步到指定群聊或平台渠道，形成内容扩散。

### 功能说明

- 当帖子发布成功后，系统支持自动同步到预设的外部渠道。
- 支持配置同步目标，例如：
  - 微信群运营渠道
  - 平台公告位
  - 其他外部内容分发渠道
- 同步内容至少包含：
  - 帖子标题 / 文案摘要
  - 封面图或缩略图
  - 跳转链接或帖子详情入口
  - 发布时间 / 发布人（按需要展示）
- 同步支持开关控制，可按学校、按活动、按帖子类型控制是否同步。

### 后台要求

- 提供同步开关与同步目标配置。
- 提供同步日志，记录同步时间、目标渠道、结果状态、失败原因。
- 同步失败时支持重试或人工补发。

### 验收标准

- 帖子发布成功后，符合条件的内容会自动触发同步。
- 管理后台可查看同步结果。
- 同步失败时可被追踪和重试。

### 当前待确认

- “群聊/平台”具体指哪些渠道。
- 是自动同步全部帖子，还是仅同步审核通过 / 指定分类 / 指定学校的帖子。
- 是否需要人工二次确认后再同步。

## 需求 4：进入小程序弹出二维码，引导用户进群

### 业务目标

- 提升社群沉淀能力。
- 为新用户提供更直接的进群入口。

### 功能说明

- 用户进入小程序首页或首次进入小程序时，弹出进群引导弹窗。
- 弹窗展示群二维码、群介绍文案、关闭按钮。
- 支持配置不同学校对应不同群二维码。
- 若当前学校有专属群，则优先展示学校专属群码；否则展示平台默认群码。
- 支持控制弹窗频率，例如：
  - 首次进入必弹
  - 每日首次弹出一次
  - 手动关闭后 N 天内不再弹出

### 前端要求

- 弹窗样式不影响主流程操作。
- 支持保存用户关闭状态，避免过度打扰。
- 支持点击查看大图、长按识别或保存二维码。

### 后台要求

- 提供群二维码配置能力。
- 支持按学校配置二维码、标题、描述文案、展示开关。

### 验收标准

- 用户进入小程序时，可按策略看到进群二维码弹窗。
- 不同学校用户可看到对应学校的二维码。
- 用户关闭后，系统按设定频率控制再次弹出。

## 需求 5：总账号可以随时增加 / 删除子管理账号

### 业务目标

- 方便平台统一管理各学校运营人员。
- 降低人工维护成本。

### 功能说明

- 总账号可在后台新增子管理账号。
- 新增时至少需要配置：
  - 登录账号
  - 登录密码或初始化密码
  - 账号状态
  - 所属学校
  - 可操作权限范围
- 总账号可编辑子管理账号信息。
- 总账号可禁用或删除子管理账号。
- 删除或禁用子管理账号后，该账号应立即失去后台访问或操作权限。

### 后台要求

- 子管理账号管理需有列表、创建、编辑、禁用、删除功能。
- 操作日志应记录总账号对后台子账号的新增、修改、删除动作。
- 建议支持重置密码功能。

### 验收标准

- 总账号可成功新增子管理账号并分配学校。
- 子管理账号创建后可正常登录，并且权限正确。
- 子管理账号被禁用/删除后无法继续管理后台。

## 需求 6：首页广告活动展示

### 业务目标

- 增强首页运营能力。
- 支持学校活动、平台活动、商业广告等内容曝光。

### 功能说明

- 小程序首页增加广告/活动展示位。
- 展示内容支持以下类型：
  - 图片广告
  - 活动 Banner
  - 学校专属活动
  - 平台通用活动
- 每条广告/活动至少支持配置：
  - 标题
  - 图片
  - 跳转类型
  - 跳转链接 / 页面路径 / 帖子 ID / 活动 ID
  - 展示开始时间
  - 展示结束时间
  - 状态（启用 / 停用）
  - 展示学校范围
  - 排序权重
- 首页优先展示当前学校适配的广告活动；若没有学校专属内容，则展示平台通用内容。

### 前端要求

- 首页广告位支持轮播或卡片展示。
- 点击广告后进入对应目标页面。
- 广告加载失败时应有降级处理，不影响首页主内容展示。

### 后台要求

- 提供广告活动管理后台。
- 支持新增、编辑、上下架、排序、学校范围配置、跳转配置。
- 建议记录曝光量和点击量，便于后续运营分析。

### 验收标准

- 首页可正常展示已启用且在有效期内的广告活动。
- 不同学校用户可看到不同学校范围的广告内容。
- 点击广告后可正确跳转。

## 四、建议补充的数据与权限设计

### 1. 学校维度

建议明确以下核心数据对象：

- 学校表
- 用户与学校关系
- 管理员与学校关系
- 帖子与学校关系
- 广告活动与学校关系
- 群二维码与学校关系

### 2. 权限维度

建议至少拆分以下权限：

- 子账号管理权限
- 帖子审核权限
- 帖子置顶权限
- 帖子删除权限
- 学校范围权限
- 广告活动管理权限
- 群二维码配置权限

## 五、开发拆分建议

建议后续按以下模块拆解开发：

- A. 学校基础数据模块
- B. 管理员学校权限模块
- C. 帖子学校归属与审核模块
- D. 小程序学校选择模块
- E. 发布同步模块
- F. 进群二维码弹窗模块
- G. 首页广告活动模块

## 六、当前仍需你确认的产品细节

以下问题建议尽快确认，否则开发时会出现分歧：

- 子管理账号是“一个账号对应一个学校”，还是“一个账号可对应多个学校”。
- 帖子审核状态流转规则是什么，例如：待审核 / 已通过 / 已拒绝。
- 置顶是全校置顶还是分类置顶，是否有时效。
- 用户切换学校后，历史帖子、收藏、点赞是否受影响。
- 发布同步的目标渠道具体有哪些，是否需要人工审核后再同步。
- 进群二维码是按学校一套，还是按活动/阶段动态切换。
- 首页广告是单一轮播位，还是多块运营位。
- 广告点击后支持哪些跳转类型。

## 七、建议的交付顺序

如果要分阶段做，建议顺序如下：

- 第一阶段：学校基础数据 + 子账号学校权限 + 帖子学校归属
- 第二阶段：基于学校的帖子审核 / 置顶 / 删除闭环
- 第三阶段：小程序学校选择 + 首页按学校展示内容
- 第四阶段：进群二维码弹窗 + 首页广告活动位
- 第五阶段：发布同步与运营日志
